1. Executive Summary

The Sentinel is a zero-trust intermediary layer purpose-built to solve the two hardest problems in enterprise NL2SQL adoption: hallucination and security. It intercepts every AI-generated SQL statement, validates it against a strict multi-gate security and semantic policy, and executes it only after all checks pass.

This project targets a critical gap in the market. Most NL2SQL demonstrations prioritize convenience over safety, allowing unchecked queries—including destructive DDL and unauthorized table access—to reach production databases. The Sentinel proves that natural-language querying and enterprise-grade governance are not mutually exclusive.

"Don't trust the LLM to write safe SQL. Trust the architecture to enforce safe SQL."

2.2 The Four Sentinel Gates

Every AI-generated SQL statement must pass through all four gates sequentially. A failure at any gate immediately terminates the pipeline and generates an audit log entry.

G1

Lexical Shape Validation

The TSqlParser decomposes the query into an abstract syntax tree. If the AST contains any DELETE, DROP, TRUNCATE, or UPDATE node, the query is instantly killed and logged as a Security Violation. Only SELECT statements proceed.

G2

Allow-List Filter

A schema-aware allow-list (defined in safe_schema.json) enumerates every permitted table, view, and column. If the generated SQL references any object outside this list—such as a Salary or UserCredentials table—the query is blocked before execution.

G3

Semantic Verification (The "Self-Critic")

A second, smaller LLM compares the original user intent against the generated SQL. It answers one question: "Does this SQL actually answer the user's question, or is it a hallucination?" This gate catches semantically valid but logically incorrect queries that pass structural checks.

G4

Execution Sandboxing

Approved queries execute against a Read-Only Named Replica with a strict 30-second timeout and an enforced TOP 1000 row limit. This prevents denial-of-service via massive, inefficient joins and guarantees that the primary production database is never impacted by AI workloads.

2.3 Implementation Details

SQL Guard Class (C# / .NET 9)

A dedicated SqlGuard class utilizes the ScriptDom TSqlParser to analyze query structure. It returns false if any DDL or DML statement other than SELECT is detected in the input string, enforcing Gate 1 at the code level.

Schema-Aware Allow-List

All permitted table and column names are maintained in a safe_schema.json configuration file. At runtime, the validation layer checks every object reference in the parsed query against this file. In production, this configuration is managed through Azure App Configuration, enabling real-time updates without redeployment.

Self-Correction Loop

If the parser detects a structural error in the generated SQL, the error message is sent back to the primary LLM with instructions to fix the query. This retry loop executes a maximum of two attempts before returning a failure to the user, preventing infinite correction cycles.

Audit Logging

Every query attempt generates an audit log entry containing: the original user prompt, the generated SQL, the gate-by-gate pass/fail status, and a timestamp. In production, these entries flow into Azure Log Analytics to power a real-time security dashboard.

3. Deployment & Hosting Strategy

3.1 Azure Infrastructure

The Sentinel is deployed as a fully serverless, zero-trust Azure architecture. Every component is provisioned via Infrastructure-as-Code (Bicep or Terraform) for repeatable, auditable deployments.

ResourceAzure ServicePurpose
Validation Layer Azure Functions (.NET 9, Isolated Worker) Stateless, serverless compute that scales instantly for query bursts
Database Sandbox Azure SQL Hyperscale — Named Replica Read-only execution environment, physically isolated from the primary
API Gateway Azure API Management (APIM) Enforces API keys, rate limiting, and throttling against DoS attacks
Secrets Management Azure Key Vault Stores connection strings and LLM API keys; accessed via Managed Identity
Configuration Azure App Configuration Hosts the safe_schema.json allow-list for real-time updates
Observability Azure Application Insights + Log Analytics Monitors all blocked queries and powers the security dashboard

Key Design Decision

The Sentinel layer never connects to the primary (write) database. It holds credentials only for the Read-Only Named Replica. Even if the application logic is compromised, no write path to production data exists.

3.2 Deployment Workflow

  1. Code-to-Cloud via GitHub Actions — Every push triggers a CI/CD pipeline that builds the .NET 9 Sentinel logic, runs unit tests against the TSqlParser gate logic (ensuring the gatekeeper cannot be tricked by SQL obfuscation), and deploys to Azure Functions.
  2. Schema Sync — The safe-table allow-list is deployed as a configuration object in Azure App Configuration, decoupling security policy updates from code deployments.
  3. Infrastructure Provisioning — A Bicep/Terraform template provisions the Azure Function, Azure SQL Named Replica, Key Vault, and all networking rules in a single declarative file.
  4. Security Monitoring — All blocked queries are streamed to Azure Log Analytics, producing a live security dashboard that shows exactly how many malicious or hallucinated queries the Sentinel has intercepted.

3.3 Security Posture

Zero Passwords in Code

The Sentinel application uses Azure Managed Identity (Entra ID) to authenticate with Key Vault and Azure SQL. No connection strings, passwords, or API keys exist in source code or configuration files.

  • Network Isolation: APIM acts as the sole ingress point; the Function App and SQL Replica are not publicly accessible.
  • Rate Limiting: APIM enforces per-key quotas to prevent prompt-injection denial-of-service attacks.
  • Row-Level Security: The database service account has RLS policies and dynamic data masking applied, providing defense-in-depth even if the allow-list is misconfigured.
  • Immutable Audit Trail: Log Analytics captures every query attempt with full provenance, supporting compliance and forensic review.
