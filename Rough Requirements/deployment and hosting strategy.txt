For Project 3: The "Sentinel" NL2SQL Governance Layer, the hosting strategy is centered on Zero-Trust Security.

Unlike Projects 1 and 2, this project acts as a "Gatekeeper." It must be deployed in a way that physically prevents unauthorized queries from reaching your data, even if the AI tries to bypass the application logic.

Document 1: Hosting & Deployment Strategy (Technical)
Project Title: The Sentinel SQL Proxy
Primary Objective: Deploy a high-security middleware layer that intercepts, parses, and validates AI-generated SQL before execution.

1. The Zero-Trust Infrastructure
Validation Layer (The Sentinel): Hosted on Azure Functions (.NET 9, isolated worker model). This provides a serverless, stateless environment that scales instantly to handle bursts of natural language queries.

The Database Tier: Azure SQL Database (Hyperscale). We will use Named Replicas as a "Sandbox."

The Strategy: The Sentinel layer never connects to the Primary (Write) database. It only has credentials for a Read-Only Named Replica.

The Network Boundary: Azure API Management (APIM). APIM acts as the front door, enforcing API keys and rate-limiting to prevent "prompt-injection" denial-of-service attacks.

The Security Vault: Azure Key Vault. All database connection strings and LLM API keys are stored here. The Sentinel app uses a Managed Identity to retrieve them—no passwords exist in the code.

2. Deployment Workflow
Code-to-Cloud: Deploy the C# Sentinel logic via GitHub Actions. The pipeline includes a step that runs Unit Tests against the TSqlParser logic to ensure the "Gatekeeper" cannot be tricked by clever SQL obfuscation.

Schema Sync: The "Safe Table List" (JSON) is deployed as a Configuration Object in Azure App Configuration, allowing you to update the "Allow-list" in real-time without redeploying code.

Auditing: All blocked queries are sent to Azure Log Analytics. This creates a "Security Dashboard" showing exactly how many "malicious" or "hallucinated" queries the Sentinel caught.
