Getting Started Guide

How to use The Sentinel to query data safely with natural language

Accessing the Application

Option 1: Interactive Demo (Recommended)

Open the demo page in your browser. The URL includes an API key that authenticates your requests: 

https://sentinelpm-func.azurewebsites.net/api/demo?code=YOUR_FUNCTION_KEY

You'll see a dark-themed interface with an input field and quick-prompt buttons.

Option 2: Direct API

Send a POST request to the query endpoint with a JSON body:

curl -X POST "https://sentinelpm-func.azurewebsites.net/api/query?code=YOUR_KEY" \
  -H "Content-Type: application/json" \
  -d '{"prompt": "Show me all orders"}'

Common Use Cases

Use Case 1: Query Your Data

Type a natural-language question into the input field and press Enter or click Query.

Example: "Show me all orders"

The Sentinel will:

  1. Send your question to Azure OpenAI, which generates: SELECT * FROM dbo.Orders
  2. Run the SQL through all four security gates
  3. Execute the approved query against the database
  4. Return the results with a full audit trail

Gate results: G1 PASS G2 PASS G3 PASS G4 PASS

Try these natural-language queries:

  • "List all customers" — returns customer names, emails, and locations
  • "Show me all products" — returns the product catalog with prices
  • "What is the average order value?" — returns an aggregate calculation
  • "Top 5 customers by total spending" — generates a JOIN with GROUP BY
  • "List all customers in Seattle" — generates a WHERE clause filter

Use Case 2: Test Security Blocking

The demo page includes red attack buttons that simulate what happens when the AI generates dangerous SQL. These demonstrate the Sentinel's core value proposition.

Example: "delete all orders" (DELETE Injection)

The demo LLM simulates generating DELETE FROM dbo.Orders. Gate 1 immediately detects the DeleteStatement in the AST and blocks execution. The self-correction loop retries twice, but if the LLM keeps generating destructive SQL, the request fails with a clear audit trail.

Gate results: G1 FAIL — "Blocked statement type: DeleteStatement"

Example: "show me salaries" (Unauthorized Table)

The LLM generates SELECT * FROM dbo.Salary. Gate 1 passes (it's a valid SELECT), but Gate 2 checks the allow-list and finds that dbo.Salary is not in the approved schema. The query is blocked before it reaches the database.

Gate results: G1 PASS G2 FAIL — "Table 'dbo.Salary' is not in the approved schema."

Example: "drop the users table" (DROP TABLE)

Gate 1 detects the DropTableStatement node in the AST. Blocked immediately.

Gate results: G1 FAIL — "Blocked statement type: DropTableStatement"

Clone and build 
git clone https://github.com/pederbenjaminmyhre/sentinel-nl2sql-governance.git
cd sentinel-nl2sql-governance
dotnet restore && dotnet build
2
Start Azurite (Azure Storage emulator, required by Azure Functions) 
npx azurite --silent
3
Start the function app (demo mode is on by default in local.settings.json) 
cd src/Sentinel.Api
func start
4
Open the demo page 
http://localhost:7071/api/demo

Or test via curl:

curl -X POST http://localhost:7071/api/query \
  -H "Content-Type: application/json" \
  -d '{"prompt": "Show me all orders"}'

Understanding the Response

Every response includes an audit object showing exactly what happened:

{
  "success": true,
  "generatedSql": "SELECT * FROM dbo.Customers",
  "results": [ { "CustomerId": 1, "FirstName": "Alice", ... } ],
  "audit": {
    "gateResults": [
      "G1-LEXICAL:PASS",
      "G2-ALLOWLIST:PASS",
      "G3-SEMANTIC:PASS",
      "G4-SANDBOX:PASS"
    ],
    "elapsedMs": 2420.06
  }
}
  • success — true if all gates passed and data was returned
  • generatedSql — the exact SQL the AI generated (visible for transparency)
  • results — the query results as an array of objects, or null if blocked
  • audit.gateResults — each gate's pass/fail status with failure reasons
  • audit.elapsedMs — total processing time in milliseconds

HTTP Status Codes

  • 200 — Query approved and executed successfully
  • 400 — Bad request (empty prompt, invalid JSON)
  • 401 — Missing or invalid function key
  • 422 — Query blocked by one or more security gates
  • 500 — Internal server error (with diagnostic details in the response body)
