Architectural Patterns

Pipeline Pattern (Gate Pipeline)

The four security gates execute as a sequential pipeline via the GatePipeline class. Each gate implements the IGate interface with a single EvaluateAsync(QueryContext) method. The pipeline iterates through gates in order and short-circuits on the first failure — if G1 blocks a DELETE, gates G2 through G4 never execute.

This pattern was chosen over parallel execution because later gates depend on earlier gate context (G2 needs the parsed AST from G1's parser), and short-circuiting prevents wasted LLM calls and database connections for already-rejected queries.

Dependency Injection & Interface Segregation

Every service is registered through .NET's built-in DI container in Program.cs. Interfaces (ILlmService, ISqlExecutor, IGate) decouple implementations from consumers, enabling the demo mode toggle — a single configuration flag swaps the entire LLM and database layer from production Azure services to in-memory mocks without changing any gate or function code.

Self-Correction Loop

When the LLM generates SQL that fails Gate 1 (lexical) or Gate 2 (allow-list), the error message is sent back to the LLM with instructions to fix the query. This retries up to 2 times before returning a failure. The loop is deliberately outside the gate pipeline — G3 and G4 failures are terminal because they indicate semantic mismatch or execution issues that the LLM cannot self-correct.

Key Design Decisions

AST Parsing, Not Regex

Gate 1 uses Microsoft's TSqlParser to decompose SQL into an abstract syntax tree, then walks the tree looking for blocked statement types. This is provably more secure than regex-based keyword detection, which can be bypassed by comment injection, Unicode tricks, or dynamic SQL construction. If the parser cannot parse it, it's blocked. If it parses to a non-SELECT node, it's blocked.

Gate 4 is Both a Gate and the Execution Step

Rather than validating and then executing in separate steps (which would require running the query twice), G4 is the execution. It opens the database connection, enforces the timeout and row limit, runs the query, and returns the results as part of its gate evaluation. This avoids a TOCTOU (time-of-check-time-of-use) vulnerability where the query could be modified between validation and execution.

Zero-Trust Network Posture

The Sentinel never holds credentials to the primary (write) database. It authenticates only to a read-only replica. Even if the application is fully compromised, no write path to production data exists. Secrets live in Key Vault, accessed via Managed Identity. No passwords, connection strings, or API keys exist in source code or configuration files.

Demo Mode as a First-Class Feature

A single Sentinel:DemoMode=true toggle swaps the LLM and database to in-memory implementations. This means recruiters, reviewers, and potential clients can test the full gate pipeline — including real TSqlParser validation and allow-list enforcement — without needing Azure OpenAI credits or a database connection. All four gates still run with real parsing logic.
