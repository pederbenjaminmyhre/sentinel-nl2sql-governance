Lessons Learned

What we discovered building the Sentinel governance layer

AI & LLM Integration

Surprise

The LLM's biggest weakness isn't hallucination — it's overcomplication

During live testing, GPT-4o-mini consistently generated syntactically correct SQL that was logically overcomplicated. When asked for "orders from last month," it produced a query with redundant MONTH() and YEAR() filters on top of a DATEADD clause. Gate 3 (semantic verification) correctly flagged this as not cleanly matching the user's intent.

Takeaway: Prompt engineering isn't enough. You need a semantic cross-check layer that compares the question to the answer, not just the SQL to the schema. This validated the entire G3 design.

Architectural Insight

The self-correction loop is more valuable than expected

We designed the self-correction loop (send the error back to the LLM for a retry) as a fallback mechanism. In practice, it became the primary way the system handles LLM imprecision. On first attempt, the LLM occasionally generates SQL with minor schema mismatches (wrong column name, missing schema prefix). The correction loop catches these at G1/G2 and fixes them transparently, so the user never sees a failure for a fixable issue.

Takeaway: Design for LLM imperfection. Build correction loops early, not as an afterthought. Cap retries tightly (we use 2) to avoid infinite loops and cost overruns.

Azure Infrastructure

Pain Point

Azure CLI SSL connection resets are persistent and unpredictable

Throughout deployment, the Azure CLI (az) suffered repeated ConnectionResetError(10054) failures for almost every command. The same command would work via curl with an explicit -4 (IPv4) flag or through PowerShell's Invoke-RestMethod with TLS 1.2 forced. The root cause was an interaction between the CLI's Python-based HTTP stack, IPv6 dual-stack networking, and the local network environment.

Takeaway: Always have a backup deployment path. We solved this by calling the Azure REST API directly with curl -4 for settings, Kudu zip deploy for code, and PowerShell for resource creation. The Bicep deployment itself was submitted via PowerShell when az failed.

Surprise

Azure OpenAI endpoint formats vary by provisioning method

The Azure OpenAI resource reported its endpoint as https://eastus2.api.cognitive.microsoft.com/, but the AzureOpenAIClient SDK expected a hostname matching *.openai.azure.com. The cognitive services endpoint works correctly — the SDK resolves it internally — but the mismatch caused hours of debugging 500 errors with no useful stack traces until we added explicit error handling.

Takeaway: Always surface the actual exception message in API responses during development. Our initial QueryFunction returned a bare 500 with no body, hiding the DNS resolution error. Adding a top-level try/catch that returns { "error": ex.Message, "type": ex.GetType().Name } instantly revealed the problem.

Pain Point

Azure subscription quotas block common architectures

The Consumption (Dynamic/Y1) hosting plan — the recommended and cheapest option for Azure Functions — was blocked by a zero-quota restriction on our Pay-As-You-Go subscription. East US 2 also rejected new SQL Server creation entirely. We initially fell back to a Basic (B1) plan at $13.14/month and moved to Central US. Days later, the quota resolved itself and we migrated to Consumption successfully.

Takeaway: Don't assume default quotas. Test deployment in your target subscription and region early. Have a fallback plan, but revisit quota restrictions periodically — they can resolve without notice.

Cost Management

Pain Point

Serverless SQL cost $9.74 in a single day

We initially deployed Azure SQL on the Serverless GP_S_Gen5_1 tier, attracted by the "auto-pause" feature that promised zero compute cost when idle. The reality was starkly different. Serverless charges $0.51/hr for vCore compute plus $0.28/hr for SQL Server licensing whenever the database is awake — a combined $0.79/hr. During a single day of deployment, testing, and debugging, every API call, health check, and connection attempt reset the 60-minute auto-pause countdown. The database stayed awake for roughly 12 hours, costing $9.74 in one day — projecting to $290/month.

Takeaway: Serverless auto-pause is designed for workloads with long idle gaps (hours between queries). For active development or portfolio projects with frequent short bursts, every connection resets the timer and you pay as if it were always-on — but at a higher hourly rate than dedicated tiers.

Win

Three tier migrations cut monthly cost by 97%

Through iterative optimization, we made three infrastructure changes that transformed the project's cost profile:

Iteration Functions SQL Est. Monthly
v1 (initial) Basic B1 ($13.14) Serverless GP_S ($9+/day) ~$284
v2 (SQL fix) Basic B1 ($13.14) Basic ($4.99) ~$19
v3 (scale-to-zero) Consumption Y1 ($0 idle) Basic ($4.99) ~$5–8

Takeaway: Cloud cost optimization is iterative, not one-and-done. Each migration was driven by real billing data, not estimates. Monitor actual spend daily during the first week of any deployment — the gap between projected and actual cost can be 10x or more. The final architecture costs 97% less than the initial deployment while delivering identical functionality.

Design & Testing

Win

Demo mode paid for itself immediately

Building demo mode (in-memory LLM + SQL mocks) as a first-class feature — not an afterthought — meant we could run the full gate pipeline locally and in CI without any Azure dependencies. All 44 unit tests pass in under 2 seconds. Local end-to-end testing works with just func start. And recruiters can evaluate the project without needing Azure credentials or incurring API costs.

Takeaway: If your project depends on external services, build a demo/mock mode from day one. It accelerates development, simplifies testing, and makes the project accessible to anyone who clones the repo.

Win

TSqlParser is the right tool for deterministic SQL validation

Using Microsoft's official T-SQL parser (ScriptDom) instead of regex or string matching proved its value immediately. Comment-obfuscated attacks like /* comment */ DROP /* comment */ TABLE Users are correctly parsed into a DropTableStatement AST node. Batch injection like SELECT 1; DROP TABLE Users is caught because the parser returns all statements in all batches. No regex pattern can match this level of reliability.

Takeaway: For security-critical parsing, always use the language's canonical parser. Regex is a vulnerability, not a solution.

Architectural Insight

The allow-list is more powerful than expected

Gate 2's schema allow-list (safe_schema.json) was designed as a simple table/column whitelist. In practice, it becomes the primary policy control surface. Security teams can add or remove queryable tables without touching code, redeploying the app, or modifying the LLM prompt. This separation of security policy from application logic is exactly what compliance teams need.

Takeaway: Externalize security policies into configuration, not code. Azure App Configuration makes this real-time and auditable.

Key Principle Reinforced

Every assumption about LLM behavior should be enforced by code, not trusted by prompt. The Sentinel exists because LLMs are statistically reliable but not deterministically safe. The gap between "usually correct" and "always safe" is where governance lives.
