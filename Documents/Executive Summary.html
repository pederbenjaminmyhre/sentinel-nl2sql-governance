Executive Summary

The Sentinel — NL2SQL Governance & Validation Layer

The Business Problem

Enterprises are rapidly adopting Natural Language to SQL (NL2SQL) tools to let non-technical users query databases conversationally. The promise is compelling: any employee can ask a question in plain English and get an instant answer from the company's data.

The reality is dangerous. Unvalidated AI-generated SQL exposes organizations to three critical risks:

  • Data Destruction — An LLM can generate DELETE, DROP TABLE, or TRUNCATE statements that destroy production data, whether through hallucination or prompt injection.
  • Unauthorized Data Access — AI may query sensitive tables (payroll, credentials, health records) that users should never see, bypassing application-level access controls.
  • Service Disruption — Poorly optimized AI-generated joins and full table scans can bring a production database to its knees, impacting all users and systems.
Most NL2SQL demos would not survive a security review. They work on stage but fail in the enterprise.

The Solution

The Sentinel is a zero-trust governance layer that sits between the AI and the database. Every AI-generated SQL statement must pass through four sequential security gates before it touches any data:

G1
Lexical Validation
Blocks destructive SQL
G2
Allow-List Filter
Only approved tables
G3
Semantic Verification
Catches hallucinations

A fourth gate enforces execution sandboxing: queries run against a read-only replica with a 30-second timeout and a 1,000-row limit, protecting production from AI-generated workloads entirely.

Business Value

For Regulated Industries (Healthcare, Finance, Legal)

The Sentinel provides the deterministic, auditable security layer that compliance teams require before approving AI access to sensitive data. Every query attempt is logged with full provenance — prompt, generated SQL, gate-by-gate pass/fail status, and timestamp — creating an immutable audit trail for regulatory review.

For Engineering & IT Leadership

The Sentinel decouples AI adoption from security risk. Teams can deploy conversational data access today without waiting for the LLM itself to become "safe enough." The architecture enforces safety at the infrastructure level, regardless of which model is used or how it is prompted.

For the Business

Self-service analytics without the risk. Non-technical users get instant answers from production data, while the Sentinel ensures they can only see what they're allowed to see and can never modify what they shouldn't touch. No DBA bottleneck. No analyst queue. No security compromise.

Key Differentiator

Unlike approaches that rely on prompting the AI to "be safe," the Sentinel uses deterministic code-level enforcement. The SQL is parsed into an abstract syntax tree and validated structurally — no amount of prompt injection or AI hallucination can bypass a parser that provably rejects non-SELECT statements. This is governance by architecture, not by hope.

"Don't trust the LLM to write safe SQL. Trust the architecture to enforce safe SQL."
